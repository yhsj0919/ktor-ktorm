# Kotlin 项目编码规范

本文档用于统一 `ktor-ktorm` 后续新增和改造代码的写作风格。

## 1. 基本原则

- 优先保持现有调用方式简单、直接、可读。
- 新增代码遵循当前分包结构，不重新引入旧的顶层业务包。
- 修改旧代码时，只在当前改造范围内统一风格，不为了格式统一进行无关重构。
- 框架代码、通用代码和业务代码保持边界清晰。
- 配置、生命周期和有状态组件不强行改写成扩展函数。

## 2. 方法实现优先级

新增方法时，优先按以下顺序判断：

1. 是否可以实现为扩展函数。
2. 是否可以实现为顶层函数。
3. 是否需要使用 `object` 保存无状态的工具能力。
4. 只有需要维护状态、依赖注入或生命周期管理时，才使用普通类或服务类。

适合使用扩展函数的场景：

- Ktor 的路由、请求和响应处理。
- `ApplicationCall`、`Route`、`RoutingContext` 的通用能力。
- Session、JSON、文件、Ktorm 实体和表的辅助操作。
- 参数转换、校验和结果转换。

不强制使用扩展函数的场景：

- Redis、数据库连接池等有状态组件。
- Koin、Ktor 插件和应用生命周期管理。
- 需要构造参数或依赖注入的服务。
- 复杂业务流程和需要明确职责的领域对象。

## 3. 基础能力注册规则

具有应用生命周期、启动初始化、资源创建、资源释放或全局状态的基础能力，优先封装为 Ktor 插件并在 `Application.module()` 中注册。

适合注册为 Ktor 插件的能力：

- 数据库连接池和数据库迁移。
- Redis 客户端和缓存连接池。
- Koin、认证、序列化、WebSocket 等框架组件。
- 启动时必须执行的初始化和应用停止时的资源释放。

插件应负责生命周期和启动失败处理，业务代码只使用已经注册好的能力，不直接创建连接池、客户端或重复执行初始化。

纯函数、无状态转换和局部辅助逻辑不需要注册插件，优先使用扩展函数或顶层函数。

除测试、命令行工具和明确的离线任务外，禁止通过静态初始化方法替代 Ktor 插件注册基础能力。

## 4. 路由扩展规则

路由方法按请求类型分组：

- `getExt`：GET 请求和 Query 参数。
- `postExt`：JSON 请求体。
- `postFormExt`：表单请求。
- `postFileExt`：Multipart 文件请求。

同一请求类型可以提供带参数和无参数两个版本，但应保持调用方式清晰。

路由扩展函数的处理顺序固定为：

1. 接收请求参数。
2. 转换为请求对象。
3. 执行参数校验。
4. 调用业务处理函数。
5. 统一包装响应。
6. 释放请求资源。

文件请求必须保证接收到的文件流在成功、异常和业务处理结束后都能释放。

## 5. 命名规则

- 类型、接口和对象使用 PascalCase。
- 普通变量、函数和扩展函数使用 camelCase。
- 常量使用 `UPPER_SNAKE_CASE`。
- 缩写按一个单词处理，例如 `Jwt`、`Http`、`Redis`、`Mysql`。
- 避免同时使用 `REDIS_HOST`、`redisHost`、`RedisHost` 表示同类配置。
- 文件名与主要类型或功能保持一致。
- 扩展函数文件使用明确的功能名称，例如 `JsonExtensions.kt`、`HttpExtensions.kt`。

## 6. 包结构规则

当前包结构按职责划分：

```text
api             HTTP API、路由、请求参数和响应对象
auth            认证、Session、JWT 和加密
common          无业务归属的通用能力
infrastructure 运行基础设施、插件、缓存和配置
persistence     数据库、Ktorm 实体和数据库扩展
service         业务服务接口及其实现
websocket       WebSocket 实体和扩展能力
```

- 根包只保留应用启动入口。
- 请求参数和响应对象放在 `api.model`。
- 数据库实体放在 `persistence.entity`。
- 数据库连接和 Ktorm 扩展放在 `persistence`。
- 异常处理统一放在 `common.error`。
- 状态页处理与异常处理放在同一包。
- Redis 配置入口和 Redis 插件实现放在 `infrastructure.cache`。

## 7. Import 规则

- 优先使用显式导入。
- 新增代码禁止使用通配符导入，例如 `import xxx.*`。
- 导入顺序为：标准库、第三方库、项目内部包。
- 同一组导入按字母顺序排列。
- 移动文件或修改包名后，必须扫描旧包路径引用。

## 8. 返回值和空值处理

- 公共函数、扩展函数和跨包调用函数明确声明返回类型。
- 私有且简单的函数可以使用类型推断。
- 对可选输入使用安全调用、默认值或明确的空值结果。
- 不使用异常代替正常的可选分支。
- 请求参数缺失时，只有业务允许时才创建默认对象。
- 不要为了省略判断而无条件使用 `!!`。

## 9. 注释、日志和异常

- 源码统一使用 UTF-8 编码。
- 公共扩展函数和公共类型使用中文 KDoc。
- 注释重点说明意图、约束和原因，不重复描述代码表面行为。
- 业务代码禁止使用 `println` 输出日志。
- 统一使用 SLF4J 日志。
- 日志和异常消息优先使用中文，并携带必要的操作上下文。
- 不在日志中输出密码、Token 等敏感信息。
- 捕获异常时只处理能够明确恢复、转换或记录的异常。

## 10. 服务和持久化

- Service 接口放在 `service`，实现放在 `service.impl`。
- Service 负责业务编排，不把 HTTP 请求解析细节放入 Service。
- Ktorm 表对象与实体接口保持成对命名：单数实体、复数表对象。
- 数据库连接通过统一入口获取，业务代码不自行创建连接池。
- 请求参数、响应对象和数据库实体不得混用。

## 11. 兼容和改造策略

- 新增能力优先提供扩展函数，避免增加不必要的工具类层级。
- 已有扩展函数的调用方式原则上保持不变。
- 重构优先调整包边界、命名和内部实现，不主动改变业务行为。
- 配置中心、动态配置等横向改造单独进行，不与普通分包重构混合提交。
- 每次结构调整后至少完成旧包引用扫描和空目录检查。
- 条件允许时执行编译和测试；无法执行时明确记录环境原因。
